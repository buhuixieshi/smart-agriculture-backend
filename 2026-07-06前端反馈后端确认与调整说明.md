## 2. 告警状态机确认

后端已冻结告警状态机。

告警状态：

```text
ACTIVE        活跃告警，未处理
ACKNOWLEDGED  已确认，但问题未结束
CLOSED        人工关闭
RECOVERED     系统自动恢复或人工恢复
```

告警严重等级：

```text
INFO
WARNING
CRITICAL
```

允许转换：

```text
ACTIVE -> ACKNOWLEDGED
ACTIVE -> CLOSED
ACTIVE -> RECOVERED
ACKNOWLEDGED -> CLOSED
ACKNOWLEDGED -> RECOVERED
```

关闭态说明：

```text
CLOSED 和 RECOVERED 都可以在前端归类为“已关闭/已结束”。
```

区别：

```text
CLOSED    更偏人工关闭
RECOVERED 更偏系统检测到恢复，例如设备重新在线
```

后端已调整：

```text
POST /api/alarms/{id}/ack 返回状态 ACKNOWLEDGED
```

旧的 `ACKED` 只作为历史兼容，不再作为新状态返回。

## 4. 害虫识别危险等级口径

已确认：

```text
害虫危险等级以后以后端本地知识库为准。
```

原因：

```text
模型主要负责识别 pestId 和 confidence。
危险等级属于业务知识库口径，应保持稳定。
```

例如：

```text
pestId=spider_mite
本地知识库 dangerLevel=HIGH
```

即使模型返回：

```json
{
  "dangerLevel": "LOW"
}
```

后端最终也会返回：

```json
{
  "dangerLevel": "HIGH"
}
```

后端已调整：

```text
pestName 优先使用本地知识库中文名
dangerLevel 优先使用本地知识库等级
confidence 仍使用模型置信度
modelStatus 仍区分 MODEL_SERVICE / MOCK_FALLBACK
```

## 5. 设备 ONLINE 状态说明

前端反馈当前没有 ONLINE 设备。

后端逻辑：

```text
设备上报 /api/iotda/report 后，会调用 updateHeartbeatByDeviceCode
对应设备 status 会更新为 ONLINE
lastHeartbeat 会更新为当前时间
```

当前设备离线大概率是因为：

```text
1. 硬件端没有持续向后端 /api/iotda/report 上报
2. 设备端 IP 或后端地址配置不一致
3. 测试期间硬件不在线
```

设备端已通知接入后端，后续硬件在线窗口需要继续验证：

```text
1. BearPi 上报是否进入 telemetry_data

3. WebSocket 是否收到 TELEMETRY 消息
4. 设备离线/恢复告警是否按预期变化
```

## 6. 灌溉超长历史记录说明

前端反馈当前灌溉数据里有一次超长历史记录，导致本月估算用水量超过月限额。

后端当前统计口径：

```text
优先使用 irrigation_record.water_amount
如果 water_amount 为空，则按 duration_seconds * 0.02 L/s 估算
```

所以如果历史记录的 `duration_seconds` 很大，会导致估算用水量很高。

当前判断：

```text
这是联调期间遗留测试数据，不是新统计口径错误。
```

前端可以先按真实返回展示。

后续如果需要清理演示数据，可以由后端统一提供 SQL 或清理接口，但本轮不直接删除历史数据，避免误删联调记录。
