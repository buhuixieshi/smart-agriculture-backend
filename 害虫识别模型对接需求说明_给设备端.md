# 害虫识别模型对接需求说明（给设备端/模型端）

## 1. 当前后端状态

后端已经实现了害虫识别接口：

```http
POST /api/pest/detect
```

前端上传图片后，后端会尝试调用一个独立的害虫识别模型服务。

但当前后端配置里模型开关是关闭的：

```yaml
pest:
  model:
    enabled: false
    url: http://127.0.0.1:5001/pest/detect
```

所以现在实际返回的是模拟识别结果：

```text
modelStatus = MOCK_FALLBACK
```


## 2. 需要设备端/模型端提供什么

请设备端或模型端提供一个可访问的 HTTP 模型服务。

后端需要的信息：

```text
1. 模型服务 IP
2. 模型服务端口
3. 模型识别接口路径
4. 是否和后端在同一台电脑运行
5. 模型接口请求格式是否支持 multipart/form-data
6. 模型返回 JSON 字段是否能按后端要求返回
7. 模型支持识别的害虫 pestId 列表
```

后端期望最终配置成类似：

```yaml
pest:
  model:
    enabled: true
    url: http://模型端IP:模型端口/pest/detect
    connect-timeout-seconds: 3
    read-timeout-seconds: 10
```

例如模型服务跑在后端电脑本机：

```yaml
pest:
  model:
    enabled: true
    url: http://127.0.0.1:5001/pest/detect
```

例如模型服务跑在设备端电脑：

```yaml
pest:
  model:
    enabled: true
    url: http://192.168.xx.xx:5001/pest/detect
```

## 3. 模型服务接口要求

### 接口地址

模型端需要提供：

```http
POST /pest/detect
```

完整地址示例：

```text
http://127.0.0.1:5001/pest/detect
```

或：

```text
http://设备端IP:5001/pest/detect
```

### 请求方式

后端会用：

```text
multipart/form-data
```

字段：

```text
file    图片文件，必填
plotId  地块 ID，可选
```

也就是模型端要能接收类似这样的请求：

```http
POST /pest/detect
Content-Type: multipart/form-data
```

```text
file = leaf.jpg
plotId = 1
```

## 4. 模型服务返回格式

模型端必须返回 JSON。

后端期望格式：

```json
{
  "pestId": "aphid",
  "pestName": "蚜虫",
  "dangerLevel": "MEDIUM",
  "confidence": 0.91,
  "message": "ok"
}
```

字段说明：

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| pestId | String | 必填 | 害虫唯一标识，后端会根据它匹配防治建议 |
| pestName | String | 建议必填 | 害虫中文名 |
| dangerLevel | String | 建议必填 | 危险等级 |
| confidence | Number | 建议必填 | 置信度，范围建议 0 到 1 |
| message | String | 可选 | 模型端补充说明 |

`dangerLevel` 建议值：

```text
LOW
MEDIUM
HIGH
```

如果模型识别失败，建议返回：

```json
{
  "pestId": "unknown",
  "pestName": "未知害虫",
  "dangerLevel": "LOW",
  "confidence": 0.0,
  "message": "cannot recognize"
}
```

## 5. 当前后端内置支持的 pestId

后端当前已经内置了这些害虫知识库：

| pestId | 中文名 | 危险等级 |
|---|---|---|
| aphid | 蚜虫 | MEDIUM |
| whitefly | 白粉虱 | MEDIUM |
| thrips | 蓟马 | MEDIUM |
| spider_mite | 红蜘蛛 | HIGH |
| armyworm | 粘虫 | HIGH |

建议模型端优先返回这些 `pestId`，这样后端可以直接匹配到防治建议。

如果模型端有新的害虫类别，也可以返回新的 `pestId`，但需要同步给后端补充 `pest_knowledge` 表数据，否则前端可能拿不到完整防治建议。

## 6. 后端调用模型的逻辑

前端上传图片到后端：

```http
POST /api/pest/detect
```

后端处理流程：

```text
1. 校验 file 是否为空
2. 如果 pest.model.enabled=true，调用模型端 /pest/detect
3. 如果模型端返回 pestId，则使用模型结果
4. 如果模型端不可访问、超时、返回异常，则自动走模拟结果
5. 根据 pestId 查询防治建议
6. 保存识别历史到 pest_detection_record
7. 返回给前端
```

如果模型服务调用成功，前端会看到：

```json
{
  "modelStatus": "MODEL_SERVICE"
}
```

如果模型服务没接上或后端仍是模拟模式，前端会看到：

```json
{
  "modelStatus": "MOCK_FALLBACK"
}
```

## 7. 设备端/模型端需要确认的问题

请设备端或模型端确认以下信息：

```text
1. 模型服务是否已经能启动？
2. 模型服务运行在哪台电脑？
3. 模型服务的 IP 和端口是多少？
4. 接口是否是 POST /pest/detect？
5. 是否支持 multipart/form-data 上传 file？
6. 返回 JSON 是否能包含 pestId、pestName、dangerLevel、confidence？
7. 支持识别哪些 pestId？
8. 模型识别一张图片大概需要多久？
9. 后端电脑能否直接访问模型服务地址？
```
