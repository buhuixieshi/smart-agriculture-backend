# 土壤湿度阈值告警说明（给前端）

## 一、功能现状

当前后端已经具备“土壤湿度低于阈值后生成告警记录”的基础能力，但它目前复用了灌溉策略表 `irrigation_strategy` 中的湿度阈值字段，不是单独的“告警阈值配置”模块。

对应用户故事：

> 作为农户，我希望能设置土壤湿度阈值，以便湿度过低时收到告警。

当前后端实现链路为：

```text
硬件/MQTT/HTTP 上传遥测数据
-> 后端保存 telemetry_data
-> 后端读取该地块 irrigation_strategy.moisture_min
-> 判断 soil_moisture 是否连续低于阈值
-> 生成 alarm 表记录，alarm_type = LOW_SOIL_MOISTURE
```

## 二、前端如何设置土壤湿度阈值

目前设置土壤湿度阈值使用灌溉策略接口：

```http
PUT /api/strategies/{plotId}
```

示例：

```json
{
  "moistureMin": 40,
  "moistureMax": 70,
  "consecutiveThreshold": 3,
  "autoMode": true,
  "maxDuration": 900,
  "cooldownMinutes": 10
}
```

字段说明：

| 字段 | 含义 |
| --- | --- |
| `moistureMin` | 土壤湿度下限，低于该值会进入低湿度判断 |
| `moistureMax` | 土壤湿度上限，主要用于自动灌溉停止判断 |
| `consecutiveThreshold` | 连续低于阈值多少次后触发告警，默认 3 |
| `autoMode` | 当前后端低湿度告警逻辑依赖该字段为 `true` |
| `maxDuration` | 自动灌溉最长持续时间 |
| `cooldownMinutes` | 自动灌溉冷却时间 |

查询当前策略：

```http
GET /api/strategies/{plotId}
```

## 三、前端如何查看低湿度告警

低湿度告警生成后，会写入 `alarm` 表。

前端可以通过告警列表接口查询：

```http
GET /api/alarms?plotId=1&alarmType=LOW_SOIL_MOISTURE
```

返回中的关键字段：

| 字段 | 含义 |
| --- | --- |
| `alarmType` | 告警类型，低湿度为 `LOW_SOIL_MOISTURE` |
| `severity` | 告警等级，可能为 `WARNING` 或 `CRITICAL` |
| `triggerValue` | 实际触发值，即当前土壤湿度 |
| `thresholdValue` | 阈值，即 `moistureMin` |
| `status` | 告警状态，常见为 `ACTIVE`、`ACKNOWLEDGED`、`CLOSED`、`RECOVERED` |
| `message` | 告警描述 |
| `createTime` | 告警生成时间 |

示例返回数据中的告警项可能类似：

```json
{
  "id": 1,
  "plotId": 1,
  "deviceId": 7,
  "alarmType": "LOW_SOIL_MOISTURE",
  "severity": "WARNING",
  "triggerValue": 36.5,
  "thresholdValue": 40,
  "status": "ACTIVE",
  "message": "Soil moisture stayed below threshold.",
  "createTime": "2026-07-09T10:30:00"
}
```

## 四、告警处理接口

确认告警：

```http
PUT /api/alarms/{id}/ack
```

关闭告警：

```http
PUT /api/alarms/{id}/close
```

恢复告警：

```http
PUT /api/alarms/{id}/recover
```

## 五、当前限制

当前功能还不是完全独立的“告警阈值管理”：

1. 低湿度阈值目前复用 `irrigation_strategy.moisture_min`。
2. 当前逻辑要求 `autoMode=true`，否则不会触发低湿度告警。
3. 默认不是一次低于阈值就告警，而是连续 `consecutiveThreshold` 次低于阈值才触发。
4. 低湿度告警目前会写入数据库，但还没有单独推送 `ALARM` 类型 WebSocket 消息给前端。
5. 前端若要实时感知，当前建议先通过 `GET /api/alarms` 定时刷新或在收到遥测数据后主动刷新告警列表。

## 六、前端建议

如果当前页面只是需要完成“设置湿度阈值并查看告警”，可以先这样对接：

1. 设置阈值页面调用 `PUT /api/strategies/{plotId}`。
2. 告警管理页面调用 `GET /api/alarms`。
3. 低湿度告警筛选 `alarmType=LOW_SOIL_MOISTURE`。
4. 告警确认/关闭分别调用对应告警处理接口。

后续如果要做真正的“收到告警通知”，后端需要再补：

```text
生成 LOW_SOIL_MOISTURE 告警后
-> 通过 WebSocket 推送 type=ALARM 的消息
-> 前端收到后弹窗/红点/消息中心提醒
```
