# 本轮前后端联调反馈修复说明

日期：2026-07-04

适用范围：本说明用于回复前端本轮联调反馈，重点说明控制命令长期停留 `SENT`、灌溉记录生成时机、硬件回执闭环，以及真实硬件设备码统一问题。

## 1. 本轮反馈结论

前端本轮验证结果中，遥测链路已经闭合：

```text
BearPi -> 后端 -> 数据库 -> REST/WebSocket -> 前端
```

已经确认真实硬件遥测可以入库，设备状态可以更新为 `ONLINE`，WebSocket 可以收到真实 `TELEMETRY` 消息。

当前唯一真实硬件设备是：

```text
deviceCode: 6a44b8fdcbb0cf6bb96ad1a1_bearpi_001
deviceType: BEARPI
status: ONLINE
```

因此，后续真实硬件控制统一使用该 BearPi 设备码。`DEV-A003 / PUMP_CONTROLLER` 当前是离线设备记录，不再作为真实水泵控制目标。

当前未闭合的问题集中在控制链路：

```text
前端 -> 后端 REST -> MQTT Broker -> BearPi -> 回执 Topic -> 后端更新状态 -> 前端查询
```

联调现象是控制命令可以进入 `SENT`，但此前超过 30 秒后仍没有 `ackAt`，说明后端已发布到 MQTT Broker，但尚未收到 BearPi 回执。

## 2. 已完成的后端修复

### 2.1 增加控制命令超时终态

新增文件：

```text
src/main/java/com/agriculture/task/ControlCommandTimeoutTask.java
```

修复内容：

```text
每 10 秒扫描一次 control_command 表。
如果命令 status = SENT，且 sent_at 超过 30 秒，且 ack_at 仍为空，
则自动更新为 TIMEOUT。
```

更新字段：

```text
status = TIMEOUT
error_message = Command reply timeout after 30 seconds.
updated_at = 当前时间
```

修复后，前端轮询：

```http
GET /api/control/commands/{commandNo}
```

不会长期只看到 `SENT`。如果 BearPi 没有回执，后端会在约 30 秒后给出明确终态：

```json
{
  "status": "TIMEOUT",
  "errorMessage": "Command reply timeout after 30 seconds."
}
```

### 2.2 灌溉记录不再按 SENT 生成

修改文件：

```text
src/main/java/com/agriculture/service/impl/ControlServiceImpl.java
```

原逻辑：

```text
MQTT 发布成功 -> command.status = SENT -> 立即创建或结束灌溉记录
```

问题：

```text
SENT 只代表命令已发到 MQTT Broker，不代表 BearPi 已经执行。
如果硬件没有动作，灌溉记录仍被生成，会误导前端。
```

修复后：

```text
ControlServiceImpl 只负责创建命令、发布 MQTT、更新 SENT/FAILED。
不再在 SENT 阶段创建或结束灌溉记录。
```

### 2.3 灌溉记录改为按硬件 SUCCESS 回执生成

修改文件：

```text
src/main/java/com/agriculture/mqtt/MqttBizMessageHandler.java
```

修复后逻辑：

```text
收到 device/control/reply 回执
-> 解析 commandId / commandNo
-> 更新 control_command.status、ack_at、error_message
-> 只有 status = SUCCESS 时，才处理灌溉记录
```

具体规则：

```text
PUMP_ON + SUCCESS  -> 创建 RUNNING 灌溉记录
PUMP_OFF + SUCCESS -> 结束最近一条 RUNNING 灌溉记录，改为 FINISHED
FAILED / TIMEOUT   -> 不生成成功灌溉记录
```

这样可以保证灌溉记录语义更接近真实 BearPi 执行结果。

```text

真实设备码：6a44b8fdcbb0cf6bb96ad1a1_bearpi_001
```

后续真实控制接口请统一使用：



```text

真实在线、真实上报遥测、真实连接硬件链路的是 BearPi。
```

前端筛选真实可控制设备时，可以临时按以下条件处理：

```text
deviceCode = 6a44b8fdcbb0cf6bb96ad1a1_bearpi_001
status = ONLINE
```

如果前端必须根据 `deviceType` 判断控制能力，当前 BearPi 的 `deviceType` 是：

```text
BEARPI
```

因此前端不要只按 `PUMP_CONTROLLER` 判断真实水泵设备，否则会把唯一真实硬件排除掉。

## 4. 当前后端控制接口

### 4.1 打开水泵

```http
POST /api/control/send?deviceCode=6a44b8fdcbb0cf6bb96ad1a1_bearpi_001&commandType=PUMP_ON&commandValue=ON
Authorization: Bearer <token>
```

### 4.2 关闭水泵

```http
POST /api/control/send?deviceCode=6a44b8fdcbb0cf6bb96ad1a1_bearpi_001&commandType=PUMP_OFF&commandValue=OFF
Authorization: Bearer <token>
```

### 4.3 查询命令最终状态

```http
GET /api/control/commands/{commandNo}
Authorization: Bearer <token>
```

最终状态以该接口为准：

```text
SUCCESS：BearPi 已回执成功
FAILED：BearPi 已回执失败，或后端发布失败
TIMEOUT：30 秒内未收到 BearPi 回执
SENT：已发到 MQTT Broker，等待 BearPi 回执
```

## 5. 后端 MQTT 下发与回执要求

后端发布控制命令 Topic：

```text
device/control/cmd
```

后端当前发布 payload 示例：

```json
{
  "commandNo": "CMD20260704103943519",
  "deviceCode": "6a44b8fdcbb0cf6bb96ad1a1_bearpi_001",
  "commandType": "PUMP_ON",
  "commandValue": "ON"
}
```

BearPi 端需要：

```text
1. 订阅 device/control/cmd。
2. 判断 deviceCode 是否等于 6a44b8fdcbb0cf6bb96ad1a1_bearpi_001。
3. 判断 commandType 是否为 PUMP_ON / PUMP_OFF。
4. 执行水泵开关。
5. 将 commandNo 原样带回回执 Topic。
```

后端订阅回执 Topic：

```text
device/control/reply
```

建议 BearPi 成功回执：

```json
{
  "commandId": "CMD20260704103943519",
  "status": "SUCCESS",
  "message": "pump on ok"
}
```

建议 BearPi 失败回执：

```json
{
  "commandId": "CMD20260704103943519",
  "status": "FAILED",
  "message": "motor timeout"
}
```

后端兼容字段：

```text
命令编号：commandId / commandNo / command_no
状态字段：status / result
消息字段：message / msg / errorMessage / error_message
```

状态兼容：

```text
OK / SUCCESS / SUCCEED / TRUE -> SUCCESS
FAIL / FAILED / ERROR / FALSE -> FAILED
```

## 6. 前端复测建议

### 6.1 无硬件回执场景

请求：

```http
POST /api/control/send?deviceCode=6a44b8fdcbb0cf6bb96ad1a1_bearpi_001&commandType=PUMP_OFF&commandValue=OFF
Authorization: Bearer <token>
```

期望：

```text
1. 初始返回 SENT。
2. 约 30 秒后查询命令状态变为 TIMEOUT。
3. ackAt 仍为空。
4. errorMessage 有超时提示。
5. 不生成成功灌溉记录。
```

### 6.2 BearPi 成功回执场景

先调用控制接口获取 `commandNo`，然后 BearPi 或 MQTTX 向：

```text
device/control/reply
```

发布：

```json
{
  "commandId": "替换为真实 commandNo",
  "status": "SUCCESS",
  "message": "pump already off"
}
```

期望：

```text
1. GET /api/control/commands/{commandNo} 返回 SUCCESS。
2. ackAt 不为空。
3. errorMessage = pump already off。
4. 如果是 PUMP_ON，则生成 RUNNING 灌溉记录。
5. 如果是 PUMP_OFF，则结束最近一条 RUNNING 灌溉记录。
```

### 6.3 BearPi 失败回执场景

向：

```text
device/control/reply
```

发布：

```json
{
  "commandId": "替换为真实 commandNo",
  "status": "FAILED",
  "message": "motor timeout"
}
```

期望：

```text
1. 命令状态变为 FAILED。
2. ackAt 不为空。
3. errorMessage = motor timeout。
4. 不生成成功灌溉记录。
```
