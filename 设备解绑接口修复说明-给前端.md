# 设备解绑接口修复说明

## 一、问题结论

前端反馈的设备解绑问题已确认并修复。

原现象：

- 调用解绑接口后，响应 `code=200`。
- 响应体里的 `plotId` 显示为 `null`。
- 但重新查询设备列表后，设备仍然绑定在原地块。

根因：

- 后端使用 MyBatis-Plus 的 `updateById` 更新设备。
- MyBatis-Plus 默认不会更新值为 `null` 的字段。
- 因此 Java 对象里的 `plotId` 已经变成 `null`，但实际 SQL 没有生成 `plot_id = null`。
- 所以数据库里的 `device.plot_id` 没有真正清空。

## 二、修复内容

后端已完成以下修复：

### 1. 旧解绑接口修复

```http
PUT /api/devices/{id}/unbind
```

现在会显式执行：

```sql
plot_id = null
```

可以真正持久化解绑结果。

### 2. 新统一绑定接口修复

```http
PUT /api/devices/binding
```

当请求体中：

```json
{
  "deviceId": 12,
  "plotId": null
}
```

或：

```json
{
  "deviceCode": "6a44b8fdcbb0cf6bb96ad1a1_bearpi_002",
  "plotId": null
}
```

后端会真正清空数据库中的 `device.plot_id`。

### 3. Entity 层兜底修复

`Device.plotId` 已增加 MyBatis-Plus 更新策略：

```java
@TableField(updateStrategy = FieldStrategy.ALWAYS)
private Long plotId;
```

这样后续即使其它更新路径需要清空 `plotId`，也不会被 MyBatis-Plus 忽略。

## 三、前端无需修改

前端当前调用方式可以保持不变。

推荐继续使用统一接口：

```http
PUT /api/devices/binding
Content-Type: application/json
Authorization: Bearer token
```

## 六、前端复测建议

### 步骤 1：先绑定设备

```http
PUT /api/devices/binding
```

```json
{
  "deviceId": 12,
  "plotId": 2
}
```

然后查询：

```http
GET /api/devices
```

确认该设备 `plotId = 2`。

### 步骤 2：解绑设备

```http
PUT /api/devices/binding
```

```json
{
  "deviceId": 12,
  "plotId": null
}
```

### 步骤 3：重新查询设备列表

```http
GET /api/devices
```

预期结果：

```json
{
  "id": 12,
  "plotId": null
}
```

即刷新页面或重新查询后，设备仍保持未绑定状态。
